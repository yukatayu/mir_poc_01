# 12 — 決定レジスタ

この文書は、主要なアーキテクチャ判断と、その現在の強度レベルを記録する。

| ID | トピック | 現在の判断 | Level | 注記 |
|---|---|---|---|---|
| D-001 | 全体アーキテクチャ | Mir、Mirrorea、PrismCascade、Typed-Effect Wiring Platform を分離しつつ相互運用可能に保つ | L1 | 中核的なアーキテクチャ上の立場 |
| D-002 | 中核実装言語 | core implementation には Rust を優先する | L2 | 推奨であり、法ではない |
| D-003 | Graph evolution | 既定の進化規則として downstream addition / leaf-style patching を優先する | L0/L1 | 強い基盤制約 |

| D-011 | Mir-0 boundary | Mir-0 を、event DAG、`place`、最小 effect request operation（ここでは `perform` と表記）、effect / contract、最小 failure space、primitive fallback、local `try` / rollback、place-local `atomic_cut`、linear resource から成る最小 kernel として固定し、`barrier`、`durable_cut`、完全な fallback normalization、`emit`、coroutine semantics、overlay alias detail は後段へ送る | L1 | Workstream A を安定化させつつ、より広い Mir / Mirrorea machinery は決めない |

| D-013 | durable_cut / barrier の責務分担 | `durable_cut` は Mir-1 側の cut vocabulary 候補とし、`barrier` は、Mir-1 に残すとしても commit-like primitive ではなく ordering primitive 候補として扱う。Mirrorea は later cut vocabulary の意味づけは行わず、その storage / distributed realization / route / audit 側だけを引き受ける | L1 | Mir-0 / Mir-1 / Mirrorea の cut 境界を明確にする |
| D-014 | durable_cut の最小 guarantee set | `durable_cut` の最小追加保証は、abstract persistence requirement を伴う durable commit guarantee である。これは successful な pre-cut prefix が local rollback / process restart / route rebinding の後に未確定へ戻らないことを要求する。具体的な storage / replication / consensus mechanism は Mir-1 では固定しない。distributed finalization は最小意味そのものではなく、Mirrorea が複数 `place` / 実ノード上で引き受ける operational realization 側に属する | L1 | Mir-1 の cut 語彙を狭く定義し、Mirrorea への責務委譲を明示する |
| D-015 | durable_cut の failure surface | durable guarantee を確立できなかった `durable_cut` は、successful な cut が成立していない失敗試行として扱う。既定分類は `Reject` とし、cut 後の外部化された obligations を明示的に巻き戻す必要がある場合だけ `Compensate` を使う。`Approximate` は contract が durability を弱めた代替結果を明示的に許す場合に限って検討し、新しい failure class は導入しない | L1 | failure lattice を再利用し、Mir-1 と Mirrorea の境界を維持する |
| D-016 | durable_cut の observation surface | successful `durable_cut` は、ある cut attempt / pre-cut prefix が persistence evidence に支持されたことを示す明示的 event が立ったときにだけ観測されたものとみなす。failed `durable_cut` は、同じ attempt に対して successful event が立つ前に failure outcome が明示されたときに観測される。audit は、cut attempt、対象 prefix、success / failure outcome、evidence の観測有無を説明できる trace を残す。evidence の具象形式は Mir-1 では固定せず、Mirrorea の実現責務に残す | L1 | 観測面の最小意味だけを Mir-1 に残し、具象表現は Mirrorea に送る |
| D-017 | durable_cut の cross-place scope rule | cross-place `durable_cut` は participating `place` ごとの local cut attempt を束ねる aggregate attempt を観測単位とし、cross-place の `prefix` は local prefix の有限対応として扱う。single-place の success / failure event は member-local observation として保持しつつ、cross-place `durable_cut` 自体は scope rule に従う aggregate success / failure event で観測する。scope rule の具体 family と evidence 形式は Mir-1 では固定せず、Mirrorea は member observation の収集・相関と aggregate event / audit representation の実現を担う | L1 | single-place semantics を壊さずに cross-place 束ねだけを最小追加する |
| D-018 | durable_cut の scope rule family | Mir-1 が現時点で必須とする cross-place scope rule profile は `all_of` のみとする。`all_of` は aggregate attempt に含まれる participating `place` のすべてについて required local observation / persistence evidence を success 条件に数える conjunction profile である。`quorum-like` は将来拡張候補として残すが、current Mir-1 には含めない。`implementation-defined` は Mir-1 の profile 語彙にはせず、Mirrorea は Mir-1 が意味づけた `all_of` などの profile について同じ aggregate success / failure 観測を保つ形で実現する裁量だけを持つ | L1 | profile family を増やしすぎず、Mir-1 の意味と Mirrorea の実現責務を切り分ける |
| D-019 | `all_of` aggregate evidence の最小要件 | `all_of` では、participating `place` の全員について、対応する local attempt / local prefix に対する success 側の local observation と、それを支える persistence evidence が outcome 判定に数えられて初めて full coverage が成立する。Mir-1 の event surface は aggregate attempt に対する full coverage 成立済みの success、またはそれ以前の明示的 failure outcome だけを意味語彙として残し、per-place evidence payload や具象表現は要求しない。audit surface は per-place の対応関係、counted local observation の有無、supporting persistence evidence が outcome 判定に数えられたかを追跡できなければならない。full coverage の不成立は success 不成立を意味するが、それだけで failure を確定しない | L1 | `all_of` の aggregate evidence を最小 coverage 条件に絞り、Mir-1 と Mirrorea の表現境界を明確にする |
| D-020 | `all_of` aggregate failure justification | `all_of` では full coverage 不成立だけでは aggregate failure を確定しない。aggregate failure event を正当化できるのは、required member の local attempt / local prefix に explicit local failure が立って同じ aggregate attempt の full coverage が不可能になった場合、または aggregate success より前にその aggregate attempt が failed outcome として明示的に閉じられた場合に限る。member-local failure event 自体は aggregate failure event の代用ではない。timeout-like budget、policy cancellation、retry exhaustion などの closure reason taxonomy は Mir-1 の独立語彙にせず、Mirrorea の operational policy / audit に残す | L1 | coverage 不足と failure 確定を分離しつつ、aggregate failure の十分条件だけを最小限で固定する |
| D-021 | `all_of` aggregate failure の audit trace 最小要件 | `all_of` の aggregate failure audit は、aggregate attempt、participating / required member 集合、各 required member と local attempt / local prefix の対応、justifying required member local failure または explicit failed closure の参照可能性、そして failure 観測時点または closure 時点の missing coverage 状態を説明できなければならない。event surface は aggregate failure outcome だけを保持し、`reason_ref` のような field 名、ID 形式、serialization、closure reason taxonomy は Mir-1 で固定せず、Mirrorea の audit representation に残す | L1 | 後から failure を説明できる最小監査情報だけを Mir-1 に残し、表現形式は Mirrorea に送る |
| D-022 | `all_of` missing coverage snapshot の基準時点と粒度 | `all_of` の missing coverage snapshot は、same aggregate attempt が aggregate failure outcome として明示的に観測された決定点を最小基準時点とする。explicit failed closure がそのまま failure outcome を明示する場合は closure 時点がこの基準時点になる。Mir-1 が要求する最小粒度は per-required-member 粒度であり、各 member について対応する local attempt / local prefix と結びつけた上で、counted success-side coverage あり、local failure により coverage 不可能、未充足の少なくとも三状態を区別できれば足りる。複数時点 snapshot、保存形式、ID 化、圧縮方式は Mir-1 で固定せず Mirrorea に残す | L1 | missing coverage の説明可能性だけを最小限で固定し、履歴管理や表現形式を Mirrorea に送る |
| D-023 | `all_of` coverage state 三状態の位置づけ | `all_of` の required member coverage state 三状態は、Mir-1 の独立した outcome / contract 語彙には上げず、aggregate failure audit と missing coverage snapshot を説明するための最小抽象区別に留める。Mir-1 が固定するのは、per-required-member で counted success-side coverage あり / local failure により coverage 不可能 / 未充足を区別可能であることだけであり、aggregate success / failure、full coverage、failure justification は従来どおり counted success-side coverage の充足と explicit failure justification で定まる。三状態の命名、field 化、representation は Mirrorea の裁量に残す | L1 | audit 説明可能性は確保しつつ Mir-1 の独立語彙を増やさない |
| D-024 | fallback / preference chain / `lease` の最小意味 | Mir-0 は従来どおり primitive fallback と monotone lifetime だけを固定し、preference chain はより広い Mir で nested fallback の正規形として扱う。fallback 付き参照または guarded reference は、1 つの論理的 access path に対する有限の guarded option chain として読み、`lease` は各 option の lifetime guard を指す最小語彙に留める。`lease` 期限切れは monotone degradation の一種であり、同じ semantic lineage で earlier option への再昇格を許さない。`try` rollback や `atomic_cut` はこの degradation order を巻き戻さない。write は current option の contract が許す場合だけ成立し、write-capable option の `lease` が期限切れた後は、後段の write-capable option への explicit fallback がない限り `Reject` とする。surface syntax、完全代数、capability lattice はなお未決定である | L2 | 既存の primitive fallback と monotone lifetime から導かれる最小意味だけを明文化し、Mir-0 を広げずに guarded option chain の解釈を与える |
| D-025 | preference chain の canonical normalization law | preference chain の canonical form は、単一の logical access path / semantic lineage に属する候補を優先順に並べた有限列であり、各 option は access target、contract surface、`lease`、capability surface を持つ。nested fallback は left-to-right flattening により canonical chain へ畳み、同じ順序を保つ nested form は同一 canonical chain を表す。canonical form は元の内側 / 外側の syntax 位置ではなく、候補の順序と guard / contract / capability の組だけを保持する。正規化は monotone degradation を保つ場合に限って適用され、`try` / rollback / `atomic_cut` はその順序を並べ替えない。write-after-expiry は canonical chain 上で earlier option へ戻さず、後段へ明示的に進むか `Reject` とする | L2 | canonical form を「正規形」に限定して固定し、完全代数や collapse rule を未決のまま残す |
| D-026 | incompatible branch の rejection phase | incompatible branch は、same logical access path / semantic lineage を共有しない branch、monotone degradation を壊す branch、または declared contract surface / capability surface だけで contract-compatible fallback successor になれない branch の総称として扱う。これらの incompatibility が declared target / lineage / contract / capability だけで判定できる場合、その nested fallback は evaluation に入る前に static rejection し、canonical chain を持つ well-formed preference chain として扱わない。逆に canonical chain 自体は well-formed であり、`lease` 期限切れ、`require` 不成立、explicit failure、write-after-expiry 後の後段不在のように current evaluation で admissible option が尽きる場合は、既存どおり dynamic `Reject` に残す。declared target / lineage / contract / capability の詳細比較粒度と mixed case の扱いはなお未決定である | L2 | canonical law を変えずに malformed branch と runtime failure を分離し、hidden repair を防ぐ |
| D-027 | fallback branch の static evidence floor | current L2 で same logical access path / semantic lineage を static に扱う最小 declared information は、各 option の `declared access target` と、fallback successor が前段 option と同じ semantic lineage を継続することを示す `documented lineage annotation` である。`declared access target` だけでは証拠として足りない。contract-compatible fallback successor を static に残す最小条件は、前段 / 後段 option の `declared capability surface` が強化されておらず、`declared contract surface` に successor 使用を明示的に打ち消す矛盾がないことである。target 不一致、lineage 否定、capability 強化、contract の明示矛盾は static rejection に寄せる。宣言不足で same-lineage も incompatibility も証明できない underdeclared case は hidden に same-path 扱いして canonical chain へ入れず、current L2 では static acceptance しない。詳細な扱いは D-028 で定める | L2 | 新しい型理論を導入せず、canonical law と rejection phase を支える最小静的証拠だけを固定する |
| D-028 | fallback underdeclared case の扱い | current L2 では、`declared access target` はあるが `documented lineage annotation` が欠ける場合、`declared access target` 自体が不足している場合、`declared contract surface` / `declared capability surface` が不足して successor 判定できない場合を underdeclared fallback case とみなし、いずれも surface-level static error とする。これらは hidden に same-path 扱いして canonical chain へ入れず、dynamic `Reject` にも落とさない。`elaboration obligation` は将来拡張候補としては残すが、current L2 の admitted path には含めない。もし将来導入する場合でも canonicalization / evaluation 前に同じ static evidence floor を満たさなければならない | L2 | hidden acceptance と hidden repair を防ぎつつ、新しい elaboration system を先回りで導入しない |
| D-029 | `documented lineage annotation` の最小 surface form | current L2 では、`documented lineage annotation` の最小 surface form を、ちょうど 1 本の fallback edge を飾る edge-local annotation として扱う。最低限必要なのは predecessor option 参照、successor option 参照、same logical access path / semantic lineage continuation の affirmative claim だけである。`declared access target` は各 option に残る別情報であり、same-lineage の static 証拠としては、両端の `declared access target` と edge-local な `documented lineage annotation` の組が必要である。annotation 単独でも target 単独でも足りない。current L2 は global lineage identifier、option-local tag、chain-level blanket annotation を要求しない。説明用記法として `lineage(A -> B)` を使ってよいが、最終 keyword / punctuation / serialization は未決定である | L2 | hidden acceptance を避けるための最小書式だけを固定し、token 設計や sugar は後段に残す |
| D-030 | representative examples の current L2 surface syntax 候補 | representative examples では、direct target に対する最小 effect request operation を `perform op on target`、option chain 経由の request を `perform op via chain_ref` と書く current L2 候補を用いてよい。option chain 側は `option name on target capability cap lease guard` による option declaration と、`chain ref = head` に続く `fallback successor @ lineage(predecessor -> successor)` による canonical-form companion notation を使ってよい。これは examples 用の安定した表記候補であり、Mir-0 の規範的 surface syntax や最終 reserved keyword を固定する判断ではない | L2 | parser 実装前に representative code の書き味を安定化するための companion syntax であり、理論本体は変更しない |
| D-031 | representative examples の `try` / `fallback` companion syntax 候補 | representative examples では、local rollback を伴う `try` を block form の `try { ... } fallback { ... }` で書く current L2 候補を用いてよい。current `place` の入れ子がすでに rollback scope を与えるので、この候補では `try` に追加の scope 指定句を要求しない。後置される `fallback { ... }` は explicit fallback branch を表し、option chain の `fallback successor` とは構文形で区別される。これは examples 用の安定した表記候補であり、`try` / `fallback` の最終 keyword、punctuation、式形式、追加 sugar を固定する判断ではない | L2 | local rollback と explicit fallback branch を representative code で安定して書くための companion notation であり、Mir-0 の意味論自体は変更しない |
| D-032 | representative examples の clause / separator / block nesting 候補 | representative examples では、`contract` を独立 block keyword に上げず、`require` / `ensure` を indented な statement-local clause として直前の `perform` に付ける current L2 候補を用いてよい。statement separator に dedicated token は要求せず、`place` / `try` / `fallback` の brace-delimited block、`perform` / `atomic_cut` / `option` / `chain` の statement line、直後の indented clause / continuation line、dedent によって最小の読みを与える。blank line は readability のためだけに用い、意味論 token としては扱わない。これは examples 用 companion notation であり、`contract` の最終 keyword 化、`contract { require { ... } ensure { ... } }` block form、`require` / `ensure` の final punctuation、explicit separator token を固定する判断ではない | L2 | parser 実装前に clause と block の読みを安定させるための companion notation であり、effect / contract semantics 自体は変更しない |
| D-033 | representative examples の `contract` surface policy | current L2 では、representative examples の `contract` を semantic role の名前に留め、request-local surface では statement-local `require` / `ensure` だけを使う。`contract { ... }` block form は将来の syntax 候補としては残すが、current L2 companion notation の optional sugar としては採用しない。これにより、`place` / `try` / `fallback` の block nesting と clause attachment の読みを 1 つに保つ。option-local admission metadata の companion notation は D-036 で別に扱う | L2 | 例示資産の読みを安定させつつ、parser-ready な keyword 設計は後段へ残す |
| D-034 | representative examples の `require` / `ensure` punctuation 候補 | current L2 では、single-line clause を `require pred` / `ensure pred` と書き、multi-line predicate が必要な場合だけ `require:` / `ensure:` に続く 1 段深い predicate block を使う。predicate block は複数 clause の列挙ではなく 1 つの predicate を複数行に折り返したものとして読み、attachment は引き続き直前の `perform` にだけ属する。predicate block は clause indent への復帰で継続行が終わり、clause suite 全体は dedent または次 statement head で終わる。これは examples 用 companion notation であり、final parser punctuation、predicate block 内の boolean expression grammar、blank line 許可は未決定である | L2 | 長い predicate でも clause attachment を安定して読めるようにしつつ、`contract` を独立 block に上げず current L2 の最小表記を維持する |
| D-035 | representative examples の predicate sublanguage 候補 | current L2 では、`require` / `ensure` にぶら下がる predicate の最小断片として、bare atom、application-like form、explicit `and`、括弧 grouping までを companion notation で使ってよい。multi-line predicate block 内の改行は continuation としてだけ読み、改行だけで implicit conjunction を導入しない。blank line も predicate block 内では許さない。これは examples 用 companion notation であり、`or` / `not` / precedence table を含む完全な predicate grammar や final parser syntax を固定する判断ではない | L2 | representative examples に必要な最小 predicate 記法だけを与え、clause attachment や `contract` surface policy を変えずに読みを安定させる |
| D-036 | representative examples の option-local declared contract surface 候補 | current L2 では、option declaration の head は引き続き `option name on target capability cap lease guard` を最小形とし、capability と `lease` だけでは option 間の admissibility の違いを書けない場合は、indented な `admit pred` / `admit:` を option-local declared contract surface の admission-side metadata として明示しなければならない。省略したまま判定へ進む branch は underdeclared 側に残る。`admit` は statement-local `require` / `ensure` を option declaration に流用しないための companion marker であり、`perform` 側の request-local clause attachment とは役割を分ける。current L2 では option-local `ensure` / `invariant` や outcome-side guarantee を導入せず、`admit` に使う predicate 断片も bare atom、application-like form、explicit `and`、括弧 grouping までの最小 fragment に留める。これは examples 用 companion notation であり、`admit` の最終 keyword / punctuation や richer な option-local outcome metadata を固定する判断ではない | L2 | option-local contract information が必要な例を書ける最小表記だけを与え、statement-local clause policy と companion notation の読みを壊さず、underdeclared handling とも整合させる |
