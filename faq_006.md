# FAQ 06

## この文書について

この文書は、2026-04-17 時点の current reading に基づいて、

- 現状で何が source-backed に分かっているか
- いま何が論点になっているか
- 何と何の比較で悩んでいるか
- その悩みが sample / prototype ではどう見えているか

を、できるだけ平易に整理した FAQ である。

規範判断の正本は `specs/`、detail-side の repository memory は `plan/`、進捗の薄い snapshot は `progress.md` と `tasks.md` にある。
この FAQ は **current explanation** であり、新しい規範判断を作る文書ではない。

## 1. まず短く言うと、いま何が分かっているのか

短く言うと、次はもうかなり分かっている。

1. `Mir` の current L2 には、**実際に走る narrow subset** がある。
2. 型システム / 定理証明 / モデル検査の線は、もう「何も無い抽象議論」ではなく、**sample-visible な bridge** を持っている。
3. `atomic_cut` / publication / handoff / visibility / rollback の線も、もう「名前だけの構想」ではなく、**比較可能な corrected prototype** を持っている。
4. いま本当に難しいのは、「何も分からない」ことではなく、**どこまでを current public surface に昇格させ、どこで止めるか** である。

つまり、難所はまだ残っているが、悩みの質は

- まだ表現も実行もできない

ではなく、

- 何を final に近いものとして昇格させるか
- 何を helper-local / bridge / reserve に留めるか

へ移っている。

## 2. すでに source-backed に分かっていること

### 2.1 runnable subset はもうある

`samples/current-l2/` の current authored corpus は、`e1` から `e23` のうち current cut に入った 16 本が runner / verification ladder に乗る。

ここでは 2 種類の「動く」がある。

- runtime まで進む sample
- static-stop まで current path で通る sample

たとえば、

- `samples/current-l2/e5-underdeclared-lineage.txt`
- `samples/current-l2/e12-underdeclared-target-missing.txt`

は「成功実行する」のではなく、**underdeclared omission を static に止められる**ことを示す source-backed sample である。

逆に prototype 側では、

- `samples/prototype/current-l2-typed-proof-model-check/p06-typed-proof-owner-handoff.txt`
- `samples/prototype/current-l2-order-handoff/p07-dice-late-join-visible-history.txt`
- `samples/prototype/current-l2-order-handoff/p08-dice-stale-reconnect-refresh.txt`

のように、runtime success まで到達しつつ、helper-local preview で theorem / model-check / order-handoff の見え方を比較できる。

### 2.2 型システム / 定理証明 / モデル検査は「全く未着手」ではない

現状で source-backed にある floor は次である。

- `formal_hook`
- `proof_notebook_review_unit`
- row-local `model_check_concrete_carriers`
- emitted artifact wiring
- sample-facing summary
- typed marker を含む corrected prototype `p06`

このため、現状の repo は

- full strong type system がまだ無い
- concrete theorem prover / model-check tool binding もまだ無い

一方で、

- typed / theorem / model-check の境界がどこか
- sample から何が見え、何がまだ helper-local か

は、かなり具体的に説明できる。

### 2.3 order / handoff / `memory_order` 再解釈も「ただのスローガン」ではない

現状で source-backed にある floor は次である。

- `atomic_cut` を local finalizing cut の nucleus に置くこと
- publication / observation / witness / finalization を分けて読むこと
- `program_order` / `dependency_order` / `publication_order` / `observation_order` / `witness_order` / `finalization_order` / `scoped_happens_before` という working relation family
- corrected prototype `p07` / `p08`

つまり repo は、もう

- 「fence っぽいものが欲しい」
- 「`memory_order` を再構築したい」

という抽象要求だけの段階ではない。

今は、

- 何を local にするか
- 何を visibility として扱うか
- 何を witness として扱うか
- 何を low-level reference family に残すか

を、sample と comparison package で切り分けている段階である。

## 3. いま論点になっていること

### 3.1 型システム / 定理証明 / モデル検査側

主な論点は次である。

1. **stronger typed surface をどこまで source に出すか**
2. **theorem discharge をどこまで public contract として見せるか**
3. **model-check の property language をどこで fixed とみなすか**
4. **concrete tool binding をいつ行うか**

ここでの悩みは、「typed / theorem / model-check を入れるかどうか」ではない。
そうではなく、

- もうある bridge floor をどう昇格させるか
- 昇格前にどの stop line を守るか

である。

### 3.2 order / handoff / `memory_order` 側

主な論点は次である。

1. **source surface を low-level family に寄せるか、relation decomposition を principal にするか**
2. **publication / observation / witness / finalization をどこまで distinct に保つか**
3. **authority-handoff と shared-memory lowering をどこまで同じ causal language で扱うか**
4. **fairness / replay の final operational profile を、mixed gate の先でどう定義するか**

ここでも悩みは、「order を研究するかどうか」ではない。
そうではなく、

- final public wording をどこまで高レベルにするか
- backend-aligned reference family をどこまで保持するか

である。

### 3.3 syntax / modality 側

ここでの主な論点は次である。

1. **見た目を compact にすること** と **semantic honesty** のどちらを優先するか
2. `lambda-circle-box` をどこまで partial basis として使うか
3. guarded lambda-calculus / Modal Dependent Type Theory / Multimodal Dependent Type Theory をどの段階で stronger candidate として昇格させるか

ここでも、いま決めるべきなのは final calculus そのものではなく、

- どの問題にどの理論 family が効いているか
- どこから先はまだ OPEN か

である。

## 4. いま何と何で悩んでいるのか

### 4.1 typed surface を早く見せるか、bridge を先に固めるか

たとえば `p06-typed-proof-owner-handoff` では、`admit` / `require` / `ensure` を含む corrected prototype が runtime success まで到達する。

ここで repo がまだ迷っているのは、

- これを stronger typed syntax の first cut として昇格させるか
- まだ sample-visible corrected prototype と helper-local preview に留めるか

である。

前者の利点は user-facing readability が上がること、後者の利点は premature な typed calculus 固定を避けられることである。

### 4.2 theorem / model-check を public contract にするか、pilot に留めるか

`p06` は theorem/model-check bridge が sample-visible になっている例だが、ここで見えているのは **final public verifier contract** ではない。

現状で repo が悩んでいるのは、

- いまの `verification_preview` / `artifact_preview` を public contract 側へ近づけるか
- まだ notebook-first / row-local carrier / helper-local preview に留めるか

である。

前者の利点は toolchain 接続がしやすくなること、後者の利点は theorem side と model-check side を premature に collapse しないことである。

### 4.3 `memory_order` exact surface に寄せるか、relation family を principal にするか

order/handoff 線で repo が今いちばん慎重なのはここである。

たとえばサイコロ所有権 handoff を考えると、

- A がサイコロを振る
- 結果を publish する
- 所有権を B に handoff する
- B がその handoff を見たとき、出目も過去に確定済みだと分かる

という読みを取りたい。

このとき、

- `acquire` / `release` / `consume` / `seq_cst` の exact family に寄せるか
- `publication_order` / `observation_order` / `witness_order` / `finalization_order` を principal wording にするか

で悩んでいる。

前者の利点は backend 参照がしやすいこと、後者の利点は distributed handoff と shared-memory lowering を同じ causal language で比較しやすいことである。

### 4.4 thread と node を同じに言い切るか、違いを残すか

repo は「thread も node も place として書きたい」という方向にかなり寄っている。
ただし、完全な同一視はしていない。

悩んでいるのは、

- surface では同じ causal language に寄せたい
- しかし lowering / evidence / transport / failure / durability では差を残したい

という 2 つを、どの wording で安全に両立させるかである。

## 5. 具体例で見ると何が分かるか

### 5.1 `e5` / `e12`: underdeclared omission は static に止められる

`e5-underdeclared-lineage` と `e12-underdeclared-target-missing` が示しているのは、

- 「宣言不足」を current source convenience cut で actual sample にできた
- しかもそれを static cluster reached として preview できる

という点である。

これは、repo が「成功例だけを並べている」のではなく、**止めるべき失敗も current path に乗せている**ことを示す。

### 5.2 `p06`: typed / theorem / model-check を sample から見られる

`p06-typed-proof-owner-handoff` が示しているのは、

- typed marker を含む corrected prototype が runtime success まで進む
- 同時に theorem/model-check bridge の preview も見える

という点である。

ただし、ここで見えているのは **full type system** でも **final theorem prover integration** でもない。
あくまで、

- type / proof / model-check の境界を sample から観察できる

ところまで current cut が進んだ、という意味である。

### 5.3 `p07`: late join visibility を sample で比較できる

`p07-dice-late-join-visible-history` が示しているのは、

- handoff 後に join した participant が
- すでに publish 済みの履歴を「過去の確定済みのもの」として読める

という current comparison line である。

これは、visibility / observation / witness の読みを、shared-space final catalog を決める前に sample で比較できることを示している。

### 5.4 `p08`: stale reconnect は rollback 後 refresh に落とせる

`p08-dice-stale-reconnect-refresh` が示しているのは、

- stale reconnect の first try が失敗し
- rollback のあと
- refresh route で回復する

という comparison line である。

ここで重要なのは、「分散 handoff が全部きれいに成功する」例だけでなく、**stale / retry / refresh の失敗経路**も sample-driven に比較していることだ。

## 7. では、いま何が「ほぼ固まっている」のか

「機能そのもの」と「その機能の final public shape」は分けて読む必要がある。

ほぼ固まっているのは次である。

- `atomic_cut` を local nucleus に置くこと
- typed / theorem / model-check を first checker cut に全部押し込まないこと
- theorem / model-check / runtime policy を分けること
- corrected runnable prototype を `samples/prototype/` に分けること
- exact rough stimulus を `samples/not_implemented/` に残すこと
- semantic honesty を syntax compactness より優先すること
- `lambda-circle-box` を partial basis candidate に留め、guarded / MDTT / MTT を stronger candidate family として保持すること

逆に、まだ固めていないのは次である。

- stronger typed surface の実昇格
- theorem discharge transport / public-contract concretization
- model-check property-language / tool-binding concretization
- final order/handoff source wording
- final shared-space fairness / replay profile

## 8. いまの repo はどこまで自律的に進められるのか

現時点の読みでは、

- current self-driven queue は `0 package`
- promoted self-driven line は close 済み
- 残っているのは mixed gate / user specification topic

である。

ただし、これは「もう何も自律的にできない」という意味ではない。
正確には、

- boundary の再整理
- comparison の追加
- corrected prototype の追加
- helper-local preview の改善

のような **比較と準備の仕事** は、まだかなり自律的に進められる。

一方で、

- final public surface への昇格
- external tool binding
- installed-binary / packaging success criteria
- shared-space final catalog

は、勝手に final adoption しない方がよい段階である。
