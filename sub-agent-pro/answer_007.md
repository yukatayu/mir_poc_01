### 1. decidable core として早めに固定してよい static judgment

早めに固定してよいのは、**局所・構造・宣言情報だけで閉じる floor** です。現状の repo で最も自然なのは、まず `same-lineage floor`、次に `missing-option structure floor`、その次に `capability strengthening floor` を first checker cut の中心に置くことです。これは current corpus と helper-local spike がすでに揃っている領域で、underdeclared branch を hidden acceptance で evaluation に流さない、という repo の fail-closed 方針とも一致します。さらに declared target missing/mismatch、edge-local lineage mismatch、missing head/predecessor/successor の stable malformed cluster は、`checked_reason_codes` を伴う narrow static cluster としてかなり早く固定してよいです。   

ただし `try` / `atomic_cut` については少し切り分けた方がよく、**structural floor 自体は decidable** でも、generic checker family や public checker row にすぐ混ぜるのはまだ早いです。repo の current cut では、これらは dedicated AST structural helper family として helper-local / test-only に置き、`place_anchor == current_place` gate や whole-store restore scope は runtime / proof boundary に残すのが自然です。要するに、**structural malformed は checker、frontier gate と restore scope は runtime/proof** です。  

`require` / `ensure` / option-local `admit`、small predicate fragment も、decidable floor としては checker-front に入れてよい候補です。ただし現段階では、request/admissibility cluster は semantic inventory としては正当でも、current fixed-subset mainline の first checker boundaryを駆動する主役ではありません。ここは parser / checker front の narrow support として進め、generic public checker contract の核にするのは後ろに置くのが後戻りが少ないです。  

### 2. undecidable / heavy なので external verifier 側へ送るべき obligation

外へ送るべきなのは、**global semantic law** と **distributed/control protocol family** です。前者の代表が `canonical_normalization_law / no_re_promotion` と `rollback_cut_non_interference / hidden_rollback_absence` で、repo 自身がこれを `residual proof obligation` と呼んでいます。ここは semantics 自体は settled でも、current checker cut に閉じ込めるべきではありません。 theorem prover 側へ送るのが自然です。  

後者の代表が、`durable_cut` の cross-place aggregate correctness、`all_of` coverage / failure justification / missing coverage snapshot、さらに shared-space の authority-serial transition、witness-aware handoff、fairness、scheduler、higher-level async-control、low-level memory-order-like family です。repo はここを明示的に重い将来 workstream として分け、`atomic_cut` を local cut の最小核に留めたまま、higher-level ordering / fairness / scheduler semantics を language core へ押し込まない current first choice を採っています。ここは theorem prover というより、**protocol verifier / model checker / out-of-band analyzer** の主戦場です。   

さらに、actual emitted handoff artifact、concrete theorem / model-check tool binding、public checker migration、consumer-specific external contract、transport protocol、host binding なども、今は external side に送るべきです。repo の現方針は、proof-obligation matrix を docs 正本に置き、boundary-specific handoff artifact や actual emitter は **concrete consumer pressure が出たときだけ reopen** する、というものです。つまり、今は “proof object を送る” のではなく、**何を送るべきかを row と evidence refs で整理する段階** です。 

### 3. current fixed-subset sample line と相性が良い boundary cut

fixed-subset sample line と最も相性が良いのは、**small decidable core + small checker floor + docs-only theorem handoff + runtime evidence producer** という cut です。言い換えると、

* source/fixture/sample は narrow static floor を machine-check
* runtime は trace / audit / formal hook を出す
* theorem side は `obligation_kind + evidence_refs` の projection bundle を受ける
* model-check side は second reserve に置く

という順序です。これは repo の current line が theorem-side first consumer を `proof_notebook` に置き、public checker や concrete model-check tool bindingを still later に残していることと整合します。   

逆に相性が悪いのは、fixed-subset sample line に **public checker contract** や **boundary-specific emitted artifact** や **concrete tool binding** を早く重ねる cut です。そうすると、source corpus / lowering / runner / verification ladder / theorem-first pilot / public surface inventory がまだ narrow 化の途中なのに、checker 出力や transport を先に凍らせることになります。repo が mixed row bundle と docs-only theorem projection を current default にしているのは、この premature freezing を避けるためです。   

### 4. full 型システムや richer verifier を見据えた minimal boundary の刻み方

後戻りを減らすには、**semantic truth / local decidable floor / external proof contract / concrete runtime evidence** を最初から別 carrier にしておくのが重要です。具体的には、language core には parser token、public CLI、host I/O、tool binding、scheduler/fairness を入れない。static checker は local verdict と stable reason family に留める。external handoff は `obligation_kind + evidence_refs` の row core から始め、`goal_text` / `proof_hint` / `consumer_hint` / actual path / URI / emitted artifact id は later attachment に残す。runtime / audit は raw trace・receipt・witness を作るが、それを public-looking checker contract に即昇格させない。この刻み方なら、将来 full 型システムや proof assistant adapter を足しても、既存の narrow floor を壊さずに extension できます。  

私なら、near-term ではこう切ります。**言語 core は `Mir-0 + current L2 の guarded-option semantics` まで。static checker は same-lineage / missing-option / capability の local floor と、必要なら separated `try`/`atomic_cut` structural helper まで。theorem prover は canonical/rollback 系の global law。model checker は `durable_cut`、shared-space、authority-serial、fairness、scheduler、memory-order family。runtime/audit は trace, witness, receipt, evidence production。** これが current repo の fixed-subset runnable path と最も噛み合い、しかも将来の richer type/proof layer への増築余地を一番きれいに残します。   

一言で圧縮すると、**“semantic meaning is small, checker discharge is local, proof is global, protocol checking is distributed, audit is evidential”** です。
