According to a document from 2026-04-13, 現在の source-backed baseline に即して答えると、`atomic_cut` の先は「1 個の memory-order 語彙群」にまとめるより、`local cut`、`commit / observation`、`authority / control-plane ordering`、`external verifier / runtime policy` の 4 層へ分ける方が自然です。current docs は、`atomic_cut` を current `place` の rollback frontier を確定する **place-local finalizing cut** に留め、C++ 的 low-level memory-order family は **immediate candidate にしない**一方で、higher-level async-control family では `authority_serial_transition_family` を first cut、`witness_aware_commit_family` を second candidate、`event_tree_execution_view` を derived execution / debug view として後段に残す整理を採っています。     

まず、あなたの 4 分割案は大筋で正しいですが、3 番目の「global commit / witness / fairness」は 1 層に束ねない方がよいです。repo 側の current reading では、global commit に当たるものは Mir-1 の `durable_cut` / aggregate attempt / scope rule (`all_of`) 側で、witness / fairness / authority ownership は shared-space / Mirrorea 側の policy family で扱う方が整理しやすいです。`barrier` も、残すなら commit-like primitive ではなく ordering primitive candidate です。つまり、`atomic_cut` から先の自然な階層は、(1) Mir-0 local cut、(2) Mir-1 commit/observation vocabulary、(3) Mirrorea/shared-space authority/fairness/control-plane family、(4) theorem/protocol/runtime-policy boundary、です。    

1 層目の **local cut** は、今の source-backed 範囲そのものです。ここで固定されているのは、`atomic_cut` が current `place` の rollback frontier を確定するだけであり、単一 process 全体の同期点でも、複数 `place` をまたぐ合意点でも、永続化完了点でもない、ということです。`try` / rollback locality も current `place` に閉じます。したがって、この層には scheduler、fairness、hardware-memory-like order を載せない方がよいです。`atomic_cut` は「最小 local cut kernel」であって、「上位 ordering の代表」ではありません。   

2 層目の **commit / observation** は Mir-1 です。ここに入るのは `durable_cut`、`barrier`（残すなら ordering primitive candidate）、cut attempt / prefix / persistence evidence / success-failure observation、cross-place aggregate attempt、scope rule、`all_of` のような profile です。この層のポイントは、「local cut を global commit に広げる」のではなく、「global observation を追加で意味づける」ことです。`durable_cut` は `atomic_cut` の単なる別名ではなく、successful な pre-cut prefix が local rollback / restart / route rebinding の後に未確定へ戻らない、という **durable commit guarantee** を加える層です。実装責務は Mirrorea に渡してよいが、意味づけ自体は Mir-1 に残す、という切り方が docs と整合します。  

3 層目の **room / authority / control-plane ordering** は、Mirrorea + shared-space 側です。ここで自然なのは、`authority-ack`、`single room authority`、`authoritative serial transition`、owner slot、delegated capability、`authority_rng` / `delegated_rng_service`、`auditable_authority_witness` などを比較する line です。重要なのは、ここで扱う ordering は「CPU の load/store order」ではなく、「誰が transition owner か」「どの段階 sequence を踏んだか」「どの witness が handoff / commit を支えたか」という **authority-aware ordering** だという点です。repo 側も、authoritative room baseline を `authority-ack + single room authority + authoritative serial transition + authority_rng` に置き、fairness trust model は `opaque authority trust` を minimal candidate、`auditable_authority_witness` を next strengthening candidate にしています。  

4 層目の **external verifier / scheduler / runtime policy** は、現在すでに `core_static_checker` / `theorem_prover_boundary` / `protocol_verifier_boundary` / `runtime_policy_boundary` の 4-way split として inventory 化されています。ここでの肝は、global ordering や fairness を core checker に押し込まず、docs 正本の proof-obligation matrix と mixed row bundle を手前に置いたうえで、concrete consumer pressure が出たときだけ handoff artifact や actual emitter を reopen することです。しかも first practical external consumer は `theorem_prover_boundary`、`protocol_verifier_boundary` は second、`runtime_policy_boundary` は later と整理されています。したがって、scheduler、timeout / retry / resend、hardware-memory-like details は最初から language core に載せるのではなく、この最外層へ送るのが current repo の line に合います。  

2 点目への答えとして、**low-level memory-order vocabulary を language core に直接持ち込まない方針は妥当**です。私は「妥当」というより、current repo の前提に対しては **ほぼ必須** だと見ます。理由は 3 つあります。第一に、source-backed な確定範囲が `atomic_cut` の place-local frontier までであり、そこからいきなり low-level order を入れると source-backed 範囲を飛び越えます。第二に、docs 自体が「low-level memory-order を早く入れると parser / checker / runtime / proof burden が同時に重くなる」と明記しています。第三に、このプロジェクトはもともと Mir / Mirrorea / PrismCascade / Typed-Effect Wiring Platform を分離して相互運用させる architecture なので、hardware-like order を core に直入れすると separation を壊しやすいです。   

そのうえで、高位 family として比較すべき first-class 候補は、私は次の 3 方向を推します。

第一候補は **`authority_serial_transition_family`** です。これは current docs が明示的に first cut に置いている family で、意味の中心は「authoritative owner が serial に transition を確定していく」ことです。比較すべき primitive は、少なくとも `authority_serial_transition_contract`、`authority_subject_ref`、`transition_kind`、`authority_owner_ref`、`authority_transition_stage_family`、`transition_stage_sequence`、`stage_local_obligation_family` / row detail、`authority_handoff_epoch_ref` です。ここで表したいのは memory barrier ではなく、**owner / stage / obligation / handoff epoch** です。これは room / control-plane ordering を表すのに、C++ `memory_order` よりはるかに自然です。  

第二候補は **`witness_aware_commit_family`** です。docs 上は second candidate ですが、fairness・audit・external verifier 接続を考えるなら非常に重要です。こちらで比較すべき primitive は、`witness_aware_commit_family` 自体に加えて、later cut としての witness row detail、`replay_attachment_ref`、`handoff_payload_ref`、そして shared-space line から見れば `auditable_authority_witness` の minimal core (`witness_kind + action_ref + draw_slot + draw_result`) です。authority placement と witness requirement、provider placement を別軸で比較できるのが強みで、fairness trust を authority trust に固定しないまま auditability を強められます。  

第三候補は **`event_tree_execution_view`** ですが、これは first-class semantics というより **derived execution / debug view** に留めるのがよいです。docs でも later candidate 扱いで、shared-space 側でも tree-like room view や leaf-to-root bubbling は source of truth ではなく derived execution / debug / explanation view に留める方が自然だとされています。したがって、これを authority-serial family と同格の semantic kernel に上げるのはまだ早い。可視化・debugger・editor support の line では有力ですが、commit/ordering の主 carrier にはしない方がよいです。 

現在の mainline に対して「今やるべきこと」と「まだ早いこと」を分けると、かなりはっきりします。**今やるべきこと**は、(a) `atomic_cut` を local cut の最小核に留めたまま、higher-level async-control family の docs-first comparison を進めること、(b) `authority_serial_transition_family` の minimal contract row、owner-slot detail、stage family / sequence を理論側 row bundle として狭く比較すること、(c) `core_static_checker` / `theorem_prover_boundary` / `protocol_verifier_boundary` / `runtime_policy_boundary` の 4-way split を維持したまま、どこまでが decidable floor でどこからが external handoff かを整理し続けること、です。これは current heavy-future workstream と open-problems の両方に一致しています。 

**まだ早いこと**は、(a) low-level memory-order-like surface を source sample や parser-facing core へ入れること、(b) scheduler / fairness / hardware-memory-like semantics を `atomic_cut` の延長として core に背負わせること、(c) public checker migration や concrete theorem/model-check production contract を前倒しすること、(d) shared-space final catalog を早く固定すること、です。current mainline はなお `Macro 4` の fixed-subset executable path と `Macro 5` の theorem/model-check bridge の early-active line にあり、`Macro 6` は docs-first boundary only、`Macro 7` は inventory plus later-gate order です。したがって、この async-control / memory-order line は **docs-only / theory-only が正しい** と読むべきです。  

sample code にいつ入れるかについては、答えは二段です。**low-level memory-order family は、今は sample core に入れない**でください。docs も「fixed subset を厚くするのはよいが、higher-level async-control family や low-level memory-order-like surface を sample core として同時に広げる段階ではない」と言っています。他方で、**high-level family 自体は sample-like に置いてよい**です。ただしその置き場所は runnable fixed subset ではなく、shared-space practical example や theorem-side docs-only row bundle です。つまり、authoritative room の概念疑似コードや authority-serial contract rows は今やってよいが、parser-free runnable fixture や source-backed sample に `memory_order` 風 surface を入れるのはまだ早い、という切り分けです。  

external verifier / model checker との境界は、私は次のように切るのが最も自然だと思います。`core_static_checker` には local / structural / decidable floor だけを入れる。`theorem_prover_boundary` には authority-serial contract、stage sequence、no hidden reorder のような **global invariant proof** を置く。`protocol_verifier_boundary` には handoff epoch、witness rows、replay attachments、payload refs、transport / quorum-like protocol propertiesを置く。`runtime_policy_boundary` には scheduler、timeout / retry / resend、fairness source placement、hardware-memory-like details を置く。これなら表現力は失わず、しかも core language が scheduler specification に変質しません。current docs でも first practical external consumer は theorem side、projection bundle は docs-only、`evidence_refs` は typed symbolic refs、public checker migration は deferred、first practical consumer class は `proof_notebook` という順序が固定されています。  

私の最終的な recommendation を一文で言うと、**`atomic_cut` の先は `memory_order` を直接持ち込むのではなく、`Mir-1 commit vocabulary` と `authority-serial / witness-aware high-level async-control family` に分解し、actual executable sample へ降ろす前に theorem/protocol/runtime-policy boundary を docs-first で切る**、です。current repo の source-backed 状態と mainline の優先順位を尊重するなら、この line が最も自然で、しかも後で scheduler や verifier を繋いでも core を壊しにくいです。
