## この文書の位置づけ

この文書は、現時点でよく誤読されやすい 4 点について、

- current repo で**すでに決まっていること**
- current repo で**まだ未決のこと**
- そのうえでの**現時点の読み**

を、正本文書に沿って整理する FAQ である。

重要なのは次の 2 点である。

1. 規範判断の正本は `specs/` であり、この文書はその要約と接続に徹する。
2. 特に 4 点目の「型体系」と「ラムダキューブ」の話は、repo がまだ final type system を固定していないので、断言できることと断言できないことを分けて書く。

## 1. `place` は server / process / 実行可能ファイルに対応するのか

### 短い答え

**現時点では、その理解は正確ではない。**

current repo での `place` は、
server や process や executable を直接指す deployment 単位ではなく、
**local state / rollback scope / ownership transfer を解釈する最小の意味論的 locus**
として定義されている。

したがって、

- `1 place = 1 process`
- `1 place = 1 executable`
- `place` ごとにビルドされた成果物が自然に相互通信する

というところまでは、まだ repo は決めていない。

### current repo で決まっていること

- `place` は Mir-0 の最小意味論コアに含まれるが、その役割は deployment 単位ではなく意味論上の境界である。
  参照: [specs/04-mir-core.md](/home/yukatayu/dev/mir_poc_01/specs/04-mir-core.md)
- `place` は「local state、rollback scope、ownership transfer が解釈される最小の locus」であり、exact routing / naming / distributed placement policy は Mir-0 の外にある。
  参照: [specs/04-mir-core.md](/home/yukatayu/dev/mir_poc_01/specs/04-mir-core.md)
- `atomic_cut` も `place` 局所の finalizing cut であって、process 全体の同期点、複数 `place` の合意点、永続化完了点ではない。
  参照: [specs/04-mir-core.md](/home/yukatayu/dev/mir_poc_01/specs/04-mir-core.md), [specs/09-invariants-and-constraints.md](/home/yukatayu/dev/mir_poc_01/specs/09-invariants-and-constraints.md)
- OS / network / process は L0 substrate 側であり、Mir Core そのものとは分けて扱う。
  参照: [specs/03-layer-model.md](/home/yukatayu/dev/mir_poc_01/specs/03-layer-model.md)

### まだ未決のこと

- exact routing
- naming policy
- distributed placement policy
- `place` と actual process / node / executable packaging の対応
- cross-place transfer の final surface syntax と static rule

これらは後段の Mir / Mirrorea 側の仕事であり、current repo はまだ固定していない。
参照: [specs/04-mir-core.md](/home/yukatayu/dev/mir_poc_01/specs/04-mir-core.md), [specs/10-open-questions.md](/home/yukatayu/dev/mir_poc_01/specs/10-open-questions.md)

### 現時点の読み

現段階では、
`place` を「最終的にどこかの process や node に実装されうる意味論上の境界」と読むのは自然だが、
**今の repo はまだ `place = process` とも `place = executable` とも言っていない**
と理解するのが最も正確である。

## 2. 実行中に patch として載せたり外したりできる system を想定しているのか

### 短い答え

**大きな方向性としては yes。ただし、無制限な hot swap を既成事実化しているわけではない。**

repo が明確に志向しているのは、

- downstream addition
- compatibility-preserving overlay
- explicit cut での patch activation
- route rebinding
- dynamic scaling / reconfiguration

であり、「実行中の system に安全に機能を重ねたり差し替えたりする」方向はかなり強い。

一方で、

- 何がどの cut で有効化されるか
- in-flight task / coroutine が patch activation や route change をまたぐときどうなるか
- runtime control plane が最終的にどういう carrier を持つか

は、まだ未決の部分が残る。

### current repo で決まっていること

- Mir Core 側では、安全な進化は first-class の設計目標であり、patch は graph discipline を尊重し、既定は downstream addition である。
  参照: [specs/04-mir-core.md](/home/yukatayu/dev/mir_poc_01/specs/04-mir-core.md)
- overlay は silent shadowing をしてはならず、互換性を保たなければならない。
  参照: [specs/04-mir-core.md](/home/yukatayu/dev/mir_poc_01/specs/04-mir-core.md), [specs/05-mirrorea-fabric.md](/home/yukatayu/dev/mir_poc_01/specs/05-mirrorea-fabric.md)
- Mirrorea は logical naming、routing、overlay insertion、patch application、audit、dynamic scaling / route rebinding を担う layer として分離されている。
  参照: [README.md](/home/yukatayu/dev/mir_poc_01/README.md), [specs/03-layer-model.md](/home/yukatayu/dev/mir_poc_01/specs/03-layer-model.md), [specs/05-mirrorea-fabric.md](/home/yukatayu/dev/mir_poc_01/specs/05-mirrorea-fabric.md)
- shared-space 側でも、membership、authoritative owner slot、delegated capability、control-plane carrier を 1 つに潰さず分ける current baseline がすでにある。
  参照: [specs/examples/121-shared-space-authoritative-room-baseline.md](/home/yukatayu/dev/mir_poc_01/specs/examples/121-shared-space-authoritative-room-baseline.md), [specs/examples/125-shared-space-control-plane-carrier-threshold.md](/home/yukatayu/dev/mir_poc_01/specs/examples/125-shared-space-control-plane-carrier-threshold.md)

### まだ未決のこと

- patch / overlay / route rebinding の full operational semantics
- in-flight coroutine / task state が route change や patch activation をまたぐときの rule
- final control-plane log schema
- final exporter / receipt schema
- final authority handoff protocol
- final distributed control algorithm

参照: [specs/04-mir-core.md](/home/yukatayu/dev/mir_poc_01/specs/04-mir-core.md), [specs/05-mirrorea-fabric.md](/home/yukatayu/dev/mir_poc_01/specs/05-mirrorea-fabric.md), [specs/10-open-questions.md](/home/yukatayu/dev/mir_poc_01/specs/10-open-questions.md), [specs/examples/125-shared-space-control-plane-carrier-threshold.md](/home/yukatayu/dev/mir_poc_01/specs/examples/125-shared-space-control-plane-carrier-threshold.md)

### 現時点の読み

「ワールドを制御している system にアバター制御を attach し、必要な capability や route が配られて動き始め、離脱時には clean に外れる」
という方向性自体は、
repo の safe evolution / overlay / delegated capability / control-plane separation の思想と整合する。

ただし、ここで注意すべきなのは、
repo の正本が今書いているのは **その方向性を可能にする意味論上の制約と layering** までであって、
最終的な運用フローや runtime 制御面まではまだ固定していない、という点である。

なお、`avatar` 自体は current repo の settled term ではなく、
近い語彙は `participant`、`membership`、`member_incarnation`、`room authority` である。

## 3. サンプルコードは実際に走る段階にあるか。自動テストはあるか。LLVM はまだ先か

### 短い答え

**限定された fixed subset については、もう走る。自動テストもある。LLVM-family backend はまだ先である。**

ただし「何でも動く」段階ではない。

**current L2 fixed subset の narrow non-production path**
である。

## 4. 型システムはラムダキューブ上のどこに属するのか

### 短い答え

**今実装している範囲については、ラムダキューブ上のどこかに置ける段階ではない。**

また、
**最終的に目指す full Mir theory 全体をラムダキューブ上の 1 点に置くのも、おそらく適切ではない。**

理由は 2 つある。

1. current 実装は、まだ typed lambda calculus として整理された core ではなく、`Program` / `Statement` / `Contract` / `Predicate` を持つ effectful operational subset だからである。
2. Mir は type system だけではなく、effect、contract、rollback、ownership / lifetime、安全な進化、operational boundary まで含むため、理論全体がラムダキューブだけで尽くせる対象ではないからである。

### まず押さえるべきこと

ラムダキューブは、
主に

- term
- type
- kind

の依存関係をどう許すかを整理する枠組みである。

一方で current repo の主眼は、

- effects
- contracts
- ownership / lifetime
- rollback / cut
- safe evolution

を含む意味論の安定化にある。

したがって、「System F より強いか」という問いに対しても、
**何を強さと呼んでいるのか**
を分けないと誤読しやすい。

### 今実装している範囲

current 実装で actual に動いている core は、
少なくとも今読める限りでは

- `Program`
- `Statement`
- `Contract(require, ensure)`
- `Predicate(atom/call/and)`

の executable subset であり、
term-level lambda、kind、universe、dependent type のようなものは current code path には出てこない。

参照: [crates/mir-semantics/src/lib.rs](/home/yukatayu/dev/mir_poc_01/crates/mir-semantics/src/lib.rs)

この意味で、**current implemented subset は System F どころか、まだ「ラムダキューブ上の typed lambda calculus」として比較する段階ではない**。

より正確には、
「effectful operational DSL と small semantic core はあるが、final type system はまだ fixed していない」
という状態である。

### 最終的に実装しようとしている範囲

ここは repo 正本に従うと、**UNRESOLVED** と答えるのが正確である。

current snapshot では、

- final type system は still later
- strong type system はまだ phase の主眼ではない
- small decidable core と theorem / protocol / runtime boundary を先に切る

という整理になっている。

参照: [Documentation.md](/home/yukatayu/dev/mir_poc_01/Documentation.md), [docs/research_abstract/phase1-current-l2-semantics-stabilization.md](/home/yukatayu/dev/mir_poc_01/docs/research_abstract/phase1-current-l2-semantics-stabilization.md), [docs/research_abstract/phase5-small-decidable-core-and-proof-boundary.md](/home/yukatayu/dev/mir_poc_01/docs/research_abstract/phase5-small-decidable-core-and-proof-boundary.md), [plan/10-roadmap-overall.md](/home/yukatayu/dev/mir_poc_01/plan/10-roadmap-overall.md)

したがって、

- `λ→`
- System F
- `λω`
- `λP`
- Calculus of Constructions

のどこに最終的に属するかを、current repo はまだ決めていない。

### 現時点の見立て

ここから先は、正本の明示判断ではなく**現時点の見立て**である。

1. 将来の **type subsystem だけ** を切り出して見ても、System F との相対的位置づけを現時点で断言することはできない。
   current repo が先に整理しているのは、ownership / lifetime / effect / contract との関係を含む wider semantics であり、type subsystem 単体の final shape はまだ fixed していないからである。
2. 逆に、**Mir 全体の理論** をラムダキューブ上の一点に置くのは無理がある。
   `specs/04` が明示する通り、Mir は type system だけではなく operational boundary と evolution rule を含むからである。

つまり、

- **今実装している範囲**
  - ラムダキューブ上の位置づけを与える段階ではない
- **最終的な full theory**
  - ラムダキューブだけで表し切る対象ではない可能性が高い
- **将来の type subsystem 単体**
  - 後で独立に整理すべきだが、current repo はまだそこを fixed していない

と答えるのが、一番誤解が少ない。

## 全体のまとめ

4 点をまとめると、現時点の current repo は次のように読むのが正確である。

1. `place` は deployment 単位ではなく、意味論上の局所境界である。
2. 実行中の patch / overlay / rebinding を許す方向性は強いが、運用詳細はまだ後段である。
3. fixed subset の sample はすでに実行・回帰確認できるが、対象はまだ narrow で、LLVM は later である。
4. 型体系はまだ final ではなく、current 実装も final target も、ラムダキューブ上に単純に置ける段階ではない。
