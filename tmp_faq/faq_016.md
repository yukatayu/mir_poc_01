## 1. 現在の一文評価

前回時点で「まだ実装が必要」と答えた範囲のうち、bounded operational alpha-0.5 / alpha-0.8 / alpha-0.9 と product alpha-1 release-candidate workflow はかなり進んだ。特に、same-session runtime carrier、typed external `AddOne` host-I/O、same-session hot-plug、session-bound devtools、product alpha CLI、local/Docker transport、local R0/R2 save/load、native host launch bundle、release check は実際に動く。

ただし、これは final public product ではない。final textual `.mir` grammar、final public API / ABI、direct Mir-to-machine-code compiler、arbitrary native package execution、WAN / federation、distributed durable save/load R3/R4、final public viewer / telemetry service は未完または non-goal として残っている。

## 2. 一番下のレイヤーは Mir 言語か

プロジェクト固有の semantic layer としては、おおむね Mir Core が一番下で合っている。ただし `specs/03-layer-model.md` 上の L0 は OS / network substrate で、Mir Core は L1 である。

したがって正確には次の読み分けになる。

- OS / socket / process / filesystem まで含めた実行基盤の最下層は、プロジェクト外部の substrate。
- Mir / Mirrorea の設計上の最下層は、Mir Core。
- 現在の product alpha で外部開発者が触る入口は、Mir Core そのものではなく、versioned `package.mir.json` と `mirrorea-alpha` CLI。

根拠:

- `specs/01-charter-and-decision-levels.md`
- `specs/02-system-overview.md`
- `specs/03-layer-model.md`
- `specs/25-product-alpha1-public-boundary.md`

## 5. Mir core に stdio はあるか

ない。これは自然で、設計通り。

`specs/23-typed-external-host-boundary.md` と AGENTS の制約では、standard I/O は Mir core primitive にしない。host input / output は typed external adapter boundary に置く。

現状できること:

- `stdio` builtin ではなく、typed external `AddOne` adapter として host-I/O を実行する。
- request / response は event DAG と observer-safe export に記録される。
- product alpha viewer からも観測できる。

現状できないこと:

- Mir core 単体の `stdin` / `stdout` builtin。
- final public interactive shell。
- product alpha path での `Hello, Taro!` string sample。
- arbitrary host adapter family。

## 6. Mir は演算能力を持つか

限定された意味では持つ。current-L2 samples は type/effect/order/model-check の semantic execution を持ち、product alpha runtime は typed external `AddOne` を実行している。

ただし、現状の Mir は汎用言語として任意のプログラムを native code に落とす段階ではない。演算能力は、repo-local sample runner、runtime carrier、typed external adapter、product alpha CLI の bounded workflow によって示されている。

`manifest.json` の要点:

## 8. バイナリは分割されるか

現状、Place ごと・node ごとに Mir program から分割 native binary が生成されるわけではない。

現在の native bundle は、基本的に 1 つの `bin/mirrorea-alpha` CLI binary と、package / devtools / reports / manifest を束ねる。Docker transport でも、`samples/product-alpha1/docker/docker-compose.product-alpha1.yml` は world service と participant service の両方で同じ `mirrorea-alpha` binary を mount して、内部 helper subcommand を別 entrypoint として起動している。

したがって現状の分割は次。

- binary split: 未実装。Mir source から複数 native binaries を生成しない。
- process/container split: product alpha Docker Compose が world / participant service を分ける。
- data/package split: `package.mir.json`、dependency packages、viewer、reports、manifest で分かれる。
- logical Place split: runtime session 内の `Place[ProductDemoRoom]`、`Place[HostAdapter]`、`ParticipantPlace[...]` などとして表現される。

どのように分けるかの現行指定は、主に次のファイルにある。

- `samples/product-alpha1/demo/package.mir.json`
- `samples/product-alpha1/demo/packages/*/package.mir.json`
- `samples/product-alpha1/docker/docker-compose.product-alpha1.yml`
- generated `manifest.json` / `launch.json`

ただしこれは final ABI / final placement compiler ではない。

## 10. 型システムを強くしたり弱くしたり、層を付け外しできるか

ただし、設計としては layer / contract / checker obligation / residual obligation を分けており、強い要求を追加するときは explicit contract update や checker line の extension として扱う。

現在動いている具体例:

- debug layer attach。
- auth layer attach。
- rate-limit layer attach。
- object package / avatar-preview package は deferred boundary evidence。
- auth / capability / membership / witness lane は route / attach decision で分離される。

## 12. 依存型はどの程度入っているか。Isabelle に近いか

現状は Isabelle に近い user-facing proof language ではない。

採っているのは、finite-index / refinement-oriented な current cut である。`specs/19-verification-stratification.md` の例では次のような indexed / refinement shape が許容される。

```text
Ref<T, region=r, cap=Read, label=L>
Message<epoch=e, incarnation=i>
Package<effects=E, failures=F>
Layer<pre=P, post=Q, effects=E, failures=F>
Cut<closed_under=causal_order>
```

現状しないこと:

- arbitrary source-term dependency。
- Mir source 内の general proof term。
- Mir source 自体を theorem prover にすること。
- Isabelle / Lean 相当の integrated proof surface。

Lean は side line として使う。今回 `python3 scripts/current_l2_lean_sample_sync.py` は成功し、`samples/lean/manifest.json` が確認された。ただしこれは final prover binding completion ではない。

## 14. atomic cut / consistent cut / save-load はどこまでできるか

`atomic_cut` は place-local rollback frontier であり、distributed commit ではない。この読みは `specs/09-invariants-and-constraints.md` と `specs/20-cut-save-load-semantics.md` に明記されている。

現在できること:

- current-L2 order-handoff sample で `atomic_cut(stage_block)` を含む handoff が動く。
- product alpha で local R0 save と bounded R2 quiescent-save が動く。
- quiescent-save は `NoInFlight`、`AllPlacesSealed`、`NoPostCutSend` を返す。

## 19. vector clock はあるか。肥大化しないか。抜ける処理はどうなっているか

現行 code では、明示的な vector clock 実装は見当たらない。現在の control は event DAG、membership epoch、member incarnation、frontier / witness / capability lane によって表現されている。

該当 code:

- `crates/mirrorea-core/src/runtime.rs`
- `crates/mirrorea-core/src/fabric.rs`
- `crates/mirrorea-core/tests/runtime_substrate.rs`

抜ける処理:

- `MembershipRegistry::mark_inactive(principal)` が membership epoch を 1 増やす。
- member record を `active = false` にする。
- member `incarnation` を 1 増やす。
- `left_at_epoch = Some(current_epoch)` を設定する。

テストで確認されていること:

- leave で epoch と incarnation が増える。
- inactive member に `left_at_epoch` がない snapshot は restore で拒否される。
- `left_at_epoch` が membership frontier を超える snapshot は restore で拒否される。
- duplicate participant は拒否される。
- current code は rejoin semantics を unresolved としており、同一 principal の再 join はそのまま許可しない。

肥大化についての現状評価:

- vector clock が実装されていないので、vector clock 自体が participant 数に比例して無限肥大化する問題は現状の implemented path にはない。
- ただし membership registry は inactive member record を保持する。長期運用での compaction / tombstone GC / rejoin policy は final ではない。
- repeated join/leave churn に対する production retention / compaction は今後の設計課題として残る。
