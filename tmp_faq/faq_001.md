## 1. 継続を multi-shot にしたいが、状態を持つ資源を保持している継続をどう弾くか

## 1-1. current repo で決まっていること

### A. 無制限 coroutine model は採らない

これはかなり明示的である。

- [specs/12-decision-register.md](/home/yukatayu/dev/mir_poc_01/specs/12-decision-register.md) の `D-008` は、
  - **structured かつ制限付きの coroutine model は妥当**
  - **無制限 coroutine model は受け入れない**
  としている。

つまり repo は最初から、

- 「継続 / coroutine は将来ありうる」
- しかし「何でも suspend / resume できる unrestricted model は採らない」

という立場である。

### B. ただし、multi-shot の exact rule はまだ未決である

[specs/10-open-questions.md](/home/yukatayu/dev/mir_poc_01/specs/10-open-questions.md) は、次を未決として明示している。

- final coroutine model:
  - one-shot と multi-shot のどちらか
  - suspension restriction
  - patch interaction
  - lifetime crossing rule
- 線形 continuation に関する既存研究を使って、制約付き multi-shot を支援するかどうか

したがって、**「multi-shot continuation を current repo がどう formalize しているか」については、まだ決まっていない**。

### C. ただし、弾くべき方向性はかなり強く示唆されている

Mir Core の不変条件はかなり強い。

- [specs/04-mir-core.md](/home/yukatayu/dev/mir_poc_01/specs/04-mir-core.md)
  - linear resource / ownership が重要
  - lifetime degradation は monotone
  - local rollback は current `place` に局所
- [specs/09-invariants-and-constraints.md](/home/yukatayu/dev/mir_poc_01/specs/09-invariants-and-constraints.md)
  - ownership / lifetime の monotonicity
  - rollback 不可能な effect は隔離または compensation が必要

したがって、repo の current 方針から自然に出るのは、

- **linear / stateful / ownershipful / rollback-sensitive なものを unrestricted multi-shot で複製してはいけない**

という方向である。

これは current current L2 の fallback 議論とも整合する。
そこでも一貫して、

- hidden duplication を認めない
- rollback で degradation order を戻さない
- ownership / lifetime は monotone

という原則を保っている。

## 1-2. 何がまだ未決か

未決なのは、主に次である。

1. multi-shot を**そもそも入れるか**
2. 入れるなら、**どの capture を許すか**
3. `place` local state、rollback frontier、host effect handle、route change 中の in-flight task をどう扱うか
4. suspension が patch / route rebinding をまたいだときの rule をどうするか

この点は [specs/05-mirrorea-fabric.md](/home/yukatayu/dev/mir_poc_01/specs/05-mirrorea-fabric.md) の

- in-flight coroutine / task state が route change や patch activation をまたぐときにどう振る舞うか

という open question ともつながっている。

つまり問題は単に「継続を何回再開できるか」ではなく、

- ownership
- lifetime
- rollback
- route / patch / execution placement

の全部にまたがる。

## 1-3. 私見

私の見解はかなりはっきりしている。

### 結論

**multi-shot continuation を入れるとしても、capture 環境は厳しく制限するべき**である。

より具体的には、

- one-shot continuation
  - linear / owned / stateful resource を capture してよい
- multi-shot continuation
  - **persistent / shareable / replay-safe** な capture だけを許す
- stateful resource
  - そのまま capture させず、
    - immutable snapshot に再化（reify）する
    - explicit token に externalize する
    - もしくは multi-shot を禁止する

のいずれかに寄せるのが自然である。

### 小さい証明スケッチ

`k` を multi-shot continuation、`r` を linear resource とする。
`k` が `r` を capture していて、`resume(k)` を 2 回できるとする。

このとき 2 通りしかない。

1. 2 回の resume が**同じ `r`** を共有する
   - すると linear / exclusive ownership が壊れる。
   - これは hidden aliasing である。

2. 2 回の resume ごとに `r` を**複製**する
   - すると linear resource duplication になる。
   - これは hidden copying である。

どちらも current repo の ownership / lifetime / rollback discipline と衝突する。

よって、**linear / stateful resource を capture した continuation を unrestricted multi-shot にしてはいけない**。

### きれいな整理

理論的にきれいにするなら、継続を次の 2 種に分けるべきである。

1. **one-shot continuation**
   - resourceful capture を許す
   - resume は高々 1 回

2. **shareable multi-shot continuation**
   - capture 環境に shareability / duplicability 制約を課す
   - `place` local mutable state や linear handle はそのまま入れない

この分け方は、repo がすでに持っている

- ownership / lifetime monotonicity
- rollback locality
- hidden duplication の禁止

と素直に整合する。

### いま plan に足すべきか

これは**今後ちゃんと考えないとまずい問題**である。
そのため今回、[plan/12-open-problems-and-risks.md](/home/yukatayu/dev/mir_poc_01/plan/12-open-problems-and-risks.md) に

- constrained continuation / multi-shot

を open problem として追加している。

## 2. 参加者が抜けた時にベクタークロックから排除するタイミング

## 2-1. current repo で決まっていること

### A. この問題は current L2 mainline ではまだ扱っていない

repo 全体を見ると、

- L1 は Mir Core
- L2 は Mirrorea Fabric
- L3 は shared space / shared state

である ([specs/03-layer-model.md](/home/yukatayu/dev/mir_poc_01/specs/03-layer-model.md))。

そして shared space 側は将来 workstream であり、[specs/11-roadmap-and-workstreams.md](/home/yukatayu/dev/mir_poc_01/specs/11-roadmap-and-workstreams.md) でも

- synchronized shared-space の小例

は Workstream G に残されている。

つまり、**participant churn を持つ shared-space causality は current L2 mainline の外側**である。

### B. 少なくとも現行 repo は vector clock policy を規範化していない

current `specs/` / `plan/` / `docs/reports/` を見る限り、

- vector clock の正式採用
- membership 変化を vector からどう除去するか
- join / leave に epoch を付けるか
- tombstone / incarnation を持つか

といった policy は**まだ規範文書に入っていない**。

言い換えると、これは repo の現時点では **OPEN** である。

### C. ただし、repo の一般原則はかなり示唆的である

repo は次を強く要求している。

- hidden acceptance をしない
- cut / boundary / audit を explicit にする
- route / patch / evolution を safe downstream addition 的に扱う
- no silent shadowing

この原則に照らすと、membership の add / remove も

- 「なんとなく局所 state から消す」

のではなく、

- **明示的な membership reconfiguration event**
- **audit 可能な configuration change**

として扱う方が自然である。

## 2-2. 何が問題か

ユーザの直感は正しい。
plain vector clock を participant 集合そのものと一体化すると、**remove が難しい**。

あなたの例を少し抽象化すると、問題はこうである。

1. ある構成 `K0 = {A, B, C}` がある
2. `C` が leave を通知する
3. `A` はそれを見て `C` を自分の表現から消す
4. しかし `B` はまだ古い構成 `K0` のまま `C` を含む message を送る
5. `A` はそれを受ける

このとき `A` は、

- 「これは old configuration の履歴なのか」
- 「新しい `C` が join したのか」

を、**plain deletion だけでは区別できない**。

## 2-3. 小さい不可能性スケッチ

member 削除を「各ノードが自分の vector からその成分を消す」だけで済ませたいとする。
しかも tombstone も epoch も incarnation も持たないとする。

すると `A` 側から見える情報は、

- 自分の local vector に `C` が無い
- 後から受けた他ノード由来 message に `C` 成分がある

だけになる。

しかしこの観測だけでは、

1. その message が**古い membership 構成**から来た
2. 新しい `C` が**再参加した**

を区別できない。

したがって、

- **deletion only**
- **no tombstone**
- **no epoch / incarnation**

の 3 つを同時に採るのは無理である。

何か 1 つは必要である。

## 2-4. 私見

私の見解は、**vector clock 本体と membership reconfiguration を分けるべき**、である。

### 結論

最も自然なのは、

1. **membership / configuration の層**
2. **causal clock の層**

を分けることである。

より具体的には、

- join / leave は ordinary message ではなく**構成変更 event**
- 構成変更は**epoch** あるいは **incarnation 付き membership** として扱う
- application message は
  - 「どの membership epoch に属するか」
  - 「その epoch 内での causal position はどこか」
  を持つ

という形が最もきれいである。

### 直感的な形

```text
membership_epoch = 7
members = {A, B, C}

-- C leaves --

membership_epoch = 8
members = {A, B}
```

このとき `epoch=7` の message に `C` が出てきても、
それは **old configuration の C** であって、
`epoch=8` の current member としての `C` ではない。

したがって `A` は、

- 「C が戻ったのか？」

ではなく、

- 「これは旧 epoch の causal history だ」

と読める。

### これがなぜ自然か

これは repo の一般原則と整合する。

#### 1. 明示的 boundary にできる

membership 変更を implicit local deletion ではなく、

- config change
- activation / deactivation
- auditable event

として扱える。

#### 2. data causality と control-plane を混ぜない

participant churn は causal data event ではなく、**configuration event** に近い。
これを vector clock の 1 成分削除に押し込むと、

- causality
- membership
- authorization

が 1 つの carrier に潰れてしまう。

#### 3. 「消すタイミング問題」が「切り替え時点問題」に変わる

plain vector clock では「いつ削除するか」が難しい。
epoch 制にすると問題は、

- いつ epoch を上げるか
- その epoch を誰が承認するか

に移る。

これは難しい問題ではあるが、少なくとも**意味の種類が明確**である。

### 「全員の参加承認後にメッセージ送信を許可する」は自然か

**かなり自然**である。

ただし厳密には、

- 「全員の同期承認が必須」

である必要はない。

本質は、

- **current membership configuration の中で有効化された participant だけが send できる**

ことを明示することである。

実装形としては複数ありうる。

1. 全員承認
2. config authority / control plane 承認
3. quorum 的承認

repo はまだここを決めていない。
ただし current principles からは、**何らかの explicit activation** は必要だと思う。

### timeout を使わずにやるなら

timeout をなるべく使わないなら、なおさら

- leave を「沈黙」ではなく明示 event にする
- join を「見かけ上の登場」ではなく activation にする
- old epoch の message を current membership と混同しない

ことが要る。

その意味では、

- plain vector clock deletion

より、

- membership epoch + per-epoch causal metadata

の方がずっと自然である。

### いま plan に足すべきか

これも**今後ちゃんと考えないとまずい問題**である。
current plan には shared-space membership / causal metadata が明示されていなかったため、今回 [plan/12-open-problems-and-risks.md](/home/yukatayu/dev/mir_poc_01/plan/12-open-problems-and-risks.md) に

- dynamic membership / causal metadata

を追加している。

ただしここで重要なのは、**vector clock を current repo が採用したわけではない**ことだ。
いま plan に足したのは、

- participant churn を持つ shared-space causality は serious open problem である

という認識だけである。

## 3. フォールバックの記法上および理論上の仕様

## 3-1. current repo で決まっていること

ここは 3 論点の中で、いちばん current repo が強く決めている。

### A. current L2 の fallback は wrapper ではない

[specs/12-decision-register.md](/home/yukatayu/dev/mir_poc_01/specs/12-decision-register.md) の `D-024` と `D-025`、および
[plan/05-fallback-lease-and-chain-semantics.md](/home/yukatayu/dev/mir_poc_01/plan/05-fallback-lease-and-chain-semantics.md)
の要点は次である。

- fallback は **guarded option chain**
- `lease` は option-local lifetime guard
- same-lineage chain は **left-to-right monotone degradation**
- **no re-promotion**
- rollback / `atomic_cut` は degradation order を巻き戻さない

つまり、current L2 は fallback を

- 「外側 wrapper が切れたら、その外へ戻る」

という読みではなく、

- 「同じ semantic lineage 上の候補列を左から右へ進む」

という読みで固定している。

### B. nested fallback は canonical chain に flatten する

`D-025` は canonical normalization law をかなり明確に述べている。

- nested fallback は left-to-right flattening
- canonical form が保持するのは候補順と guard / contract / capability
- 元の outer / inner syntax 位置は保持しない

したがって、あなたの例

```text
d = (e -[fb]-> b)
c = (d -[fb]-> a)
```

を current L2 的に読むなら、same-lineage / monotone degradation が成り立つ限り、

```text
e > b > a
```

へ正規化して読むのが基本である。

### C. したがって current repo の答えは「基本は b、その後に a」である

上の canonical chain では、

1. `e` を試す
2. `e` が non-admissible / explicit failure なら `b`
3. `b` も使えなければ `a`

である。

つまり、**current repo の reading では、外側 `d` の見た目だけを理由に `a` へ飛ばない**。

## 3-2. 小さい証明スケッチ

current L2 は、

```text
canon(fallback(x, y)) = canon(x) ++ canon(y)
```

という最小 law を採っている。
ただし same-lineage / monotone degradation が成り立つ場合に限る。

そこで、

```text
d = fallback(e, b)
c = fallback(d, a)
```

とすると、

```text
canon(d) = [e, b]
canon(c) = canon(d) ++ [a] = [e, b, a]
```

になる。

したがって `e` が脱落した直後に参照すべき候補は `b` であり、
`a` は `b` まで尽きた後段候補である。

これが current repo の canonical chain reading である。

## 3-3. `a` に行くように「寿命の伝播を止める」構文は自然か

### current repo の答え

**current repo はその構文を採っていない。**

しかも私は、fallback 本体にそれを入れるのは**あまり自然ではない**と考える。

### 理由

それは 2 つの異なるものを混ぜるからである。

1. **same-lineage chain の後段 degradation**
2. **outer condition による chain selection**

`e > b > a` は 1 である。
しかし

- 「`d` が死んだら `a` に飛ぶ」

は、2 に近い。

つまり意味としては、

- chain 1: `e > b`
- chain 2: `a`
- outer condition が生きている間は chain 1、そうでなければ chain 2

である。

これは fallback の 1 本の canonical chain ではなく、**chain の外側での選択**である。

### `<...>` のような stop 構文の問題

たとえば

```text
d = <(e -[fb]-> b)>
c = (d -[fb]-> a)
```

や

```text
c = (<d> -[fb]-> a)
```

のような構文を入れると、一見柔軟ではある。
しかし理論的には、次の 2 つが曖昧になる。

1. これは canonical chain の内部規則なのか
2. それとも chain 間選択の sugar なのか

もし 1 として扱うと、

- canonical normalization law が濁る
- no re-promotion の境界が見えにくくなる
- outer/inner wrapper 誤読が復活する

ので、current repo がここまで避けてきた drift source を再導入しやすい。

## 3-4. 私見

私の見解は次である。

### 結論

**fallback 本体には stop 構文を足さず、必要なら別の selection / elaboration construct に出す方がきれい**である。

### きれいな分け方

1. **fallback chain**
   - same-lineage degradation を表す
   - current L2 の `e > b > a`

2. **selection / specialization / elaboration**
   - outer condition に応じてどの chain を採るかを決める
   - static に分かるなら compile-time / elaboration
   - dynamic にしか分からないなら runtime selection

この切り分けなら、

- fallback は fallback のまま
- outer condition は outer condition のまま

で混線しない。

### 「自然な答えは a」という感覚について

この感覚は分かる。
特に

- 変数 `d` の寿命
- `d` が指している中身の chain

を別物として見るなら、

- `d` という container が死んだら `a`

と考えたくなる。

ただし current repo は現段階では、

- variable lifetime
- referent chain lifetime

を別の formal layerに切り分けて fallback semantics を作っていない。
current current L2 で正本なのは、

- option-local `lease`
- canonical same-lineage chain

の方である。

したがって、友人の

- 「`d` そのものの寿命と `*d` の寿命を区別すると自然に `a` では？」

という指摘は、**将来の typed/static layer では十分ありうる見方**だが、
current current L2 の fallback semantics をそのまま置き換える理由にはならない。

### compile-time と runtime をどう分けるべきか

ここは以前の議論ともつながる。
私の考えでは、きれいな形は次である。

- **static / type layer**
  - chain の well-formedness
  - どこまで pruning / specialization できるか
  - static に outer condition が決まるなら chain selection を先に行う

- **runtime layer**
  - static に残った chain に対して admissibility を順に評価する

つまり、

- 「`a` に行くか `b` に行くか」を compile-time に決められる条件なら elaboration
- 実行時にしか決まらない条件なら selection
- fallback 本体は same-lineage degradation のみに使う

という分担である。

これはかなり理論的にきれいで、repo の

- semantics と helper convenience を分ける
- hidden acceptance を認めない
- canonical chain を保つ

という原則にもよく合う。

### 現行仕様を変える必要があるか

**私は、現時点では変える必要はない**と考える。

理由は次である。

1. current repo はすでに fallback を guarded option chain としてかなり強く固定している
2. regression fixtures と prose / notation もその方向で揃っている
3. ここで stop 構文を fallback 本体へ入れると、むしろ chain semantics を再び濁らせる

したがって、

- current fallback semantics は維持
- outer condition で別 chain に切り替えたい問題は、将来の selection / elaboration workstream として別に扱う

のが最小である。

## 4. 最後に短くまとめる

### 継続 / multi-shot

- current repo は **restricted coroutine はありうるが unrestricted は不可** という立場である。
- multi-shot の exact rule は未決。
- 私見では、multi-shot は capture 制約つきにすべきで、stateful / linear resource をそのまま捕まえる continuation は弾くべきである。

### participant churn と vector clock

- current repo はまだ vector clock / membership policy を決めていない。
- これは shared-space / Mirrorea future work である。
- 私見では、plain vector deletion はつらく、**membership epoch / configuration change と causal clock を分ける**のが最も自然である。

### fallback 記法 / 理論

- current repo は fallback を **guarded option chain** として固定している。
- nested fallback は canonical chain に flatten する。
- あなたの例では current repo の答えは **基本は `b`、`b` も尽きたら `a`** である。
- `a` に飛ぶための stop 構文を fallback 本体へ入れるより、将来の selection / elaboration construct として分ける方がきれいだと考える。
