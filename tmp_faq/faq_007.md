規範判断の正本は `specs/`、長期の repository memory は `plan/`、薄い snapshot は `progress.md` と `tasks.md` にある。

## 1. 先に短く答えるとどうか

短く言うと、次の読みが正確である。

1. **かなり進んでいる。**
   parser-free current L2、compile-ready minimal actualization、fixed-subset source sample、corrected prototype、helper-local compare floor は既にあり、実装と実行による比較検証もかなり進んでいる。
2. **しかし、二大問題を完全に解決したとはまだ言えない。**
   source-backed に固まっているのは、あくまで
   - current first line
   - retained alternatives
   - stop line
   - mixed gate
   - true user-spec gate
   の切り分けである。
3. **言語側の実装まで終わっているともまだ言えない。**
   current repo が持っているのは、final parser / final public checker / final runtime API ではなく、
   parser-free validation substrate と fixed runnable subset、bridge-floor prototype、helper-local preview / pre-floor である。
4. **現在地は「何も無い抽象研究」ではない。**
   むしろ、
   - theory-lab の first-line integration は close 済み
   - narrow mixed-gate pre-floor も close 済み
   - remaining work は mixed gate と user-spec gate に narrowed
   という段階である。
5. **「これらに答えればもう最後まで完全自走できるか」には、厳密には yes と言えない。**
   true user-spec gate に答えればかなり先まで自走しやすくなるが、remaining mixed gate は単なる好みではなく、実装と検証で再確認しながら actual adoption を決める必要がある。

## 5. 「二大問題を完全に解決し、言語側の実装まで終える」ために残っているもの

今後の残りは、大きく 3 層に分かれる。

| emitted artifact / handoff contract | family-first symbolic-ref ratchet | final schema をどう actualize するか |

これは repo 内比較だけでは final に決められず、

ただし、理論構築が完全に終わったわけではない。

これは「白紙からの大理論」ではないが、
**still substantial** である。

特に重要なのは、
この残りが mere documentation cleanup ではなく、
**implementation / validation / compare evidence を見ながら actual adoption を切る類の判断**
だという点である。

## 7. では、何を答えればどこまで自走できるのか

### 7.1 先に結論

**true user-spec gate に答えれば、repo はかなり先まで自走しやすくなる。**
しかし、
**それだけで「最後まで完全に何も聞かずに自走できる」とは、まだ約束しにくい。**

理由は 2 つある。

1. remaining mixed gate は user preference だけではなく、implementation evidence に応じて actual adoption を切る engineering judgment だから。
2. final parser / public contract / concrete tool binding / application realization は、selected target に依存して package 分解自体が変わるから。

したがって、正確な答えは次になる。

- **かなり長く自走できるようにはなる。**
- **だが、最後まで完全 no-question を保証するのはまだ無理である。**

これは pessimism ではなく、現在の source-backed docs がそう読めるからである。

### 7.2 それでも、今答えてもらえると一番効くもの

もし「ここでまとめて答えて、その後はできるだけ長く自走してほしい」のであれば、
少なくとも次が有効である。

#### A. true user-spec gate

1. **first external integration target は何か**
   - theorem prover first
   - model-check first
   - host / app first
2. **backend / tooling success criteria は何か**
   - repo-local CLI が通ればよいか
   - installed binary まで要るか
   - host adapter / FFI まで要るか
3. **first application target は何か**
   - shared-space room
   - theorem consumer
   - model-check workflow
   - 他の host/application
4. **shared-space final catalog の scope はどこまでか**
5. **fairness / replay final operational profile をどこまで要求するか**

#### B. mixed gate を進めやすくする preference

1. **stronger typed surface は早めに source へ上げたいか**
   それとも checker-adjacent principal を長く保ちたいか。
2. **theorem discharge を public-facing contract に寄せたいか**
   それとも notebook-first / row-local line を長く保ちたいか。
3. **model-check property language は早めに固定したいか**
   それとも carrier / seam compare を長く保ちたいか。
4. **order / handoff wording は high-level relation decomposition を principal にしたいか**
   それとも low-level family 寄りの wording を強めたいか。

この B は true user-spec gate ではない。
しかし、design preference を先に知っておくと reopen 回数を減らせる。

## 8. もし「これに答えればもう最後まで自走できる」が理想なら、現実にはどうか

### 8.1 楽観的に言えること

次まではかなり現実的に言える。

- current docs-first theory line の remaining mixed gate をさらに narrowing する
- first concrete target に向けて package を切り直す
- parser/public surface/tool binding の implementation package を始める
- representative corpus と compare floor を拡張しながら ratchet 方式で進める

つまり、
**「答えをもらえれば、今よりはるかに長い距離を user 追加入力なしで進める」**
ことはできる。

### 8.2 正確を優先すると言えないこと

しかし、次はまだ言えない。

- 「この FAQ に答えれば、そのまま final parser / final public contract / final tool binding / final app completion まで 100% 無停止で行ける」

根拠は明確である。

1. `specs/examples/462` は remaining work を mixed gate / user-spec gate に narrowed したと読むが、final adoption 済みとは読んでいない。
2. `specs/examples/463`、`464`、`465` は compare floor を厚くしたが、public contract adoption とは明示的に書いていない。
3. `Documentation.md`、`progress.md`、`tasks.md` は final parser grammar、final public verifier contract、concrete tool binding、backend / app target を still open としている。
4. `plan/13` と `plan/17` は、heavy future workstream と autonomy gate をまだ mixed / user-spec required として残している。

したがって、
**「もう最後まで完全自走」ではなく、
「final adoption / target concretization の手前まではほぼ整理済みで、その先もかなり長く自走できるが、最後の数個の gate はまだ実装 evidence と target choice に依存する」**
が正確である。
