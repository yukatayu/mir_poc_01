## 3. 言語機能ごとに、どの段階で見ればよいのか

### 結論

各 feature family は、単に「設計済み / 未実装」ではなく、
少なくとも次の **maturity stage** で見るのが自然である。

| Stage | 意味 |
|---|---|
| `S0` | 要求探索 |
| `S1` | 理論骨格 |
| `S2` | 論理学 / 計算機科学 / invariant 整理 |
| `S3` | current spec / boundary convergence |
| `S4` | narrow implementation |
| `S5` | executable / validation ratchet |
| `S6` | sample / human docs integration |

重要なのは、これは strict waterfall ではないことだ。

- ある feature は `S5` まで進んでいる
- 別の feature は `S2` にいる
- 同じ feature でも、docs-first boundary は `S3`、actual runtime は `S1`

## 4. 「理論で裏付けつつ実装し、きちんと動くことを確かめながら進む」という意図は共有しているか

### 短い答え

**共有している。**

しかも current repo は、すでにその方向へかなり強く寄っている。

repo の進め方は、少なくとも今のところ次の順で読むのが自然である。

1. semantics / invariants / boundary を docs-first で切る
2. narrow implementation を入れる
3. parser-free fixture / sample / regression / smoke で ratchet する
4. formal boundary / proof obligation wording を並走で揃える
5. sample / README / FAQ / progress / tasks を同期する

これは

- 理論だけ先に置いて implementation を先送りにする
- implementation だけ先に広げて docs / invariants を後追いにする

のどちらでもない。

### 私の見解

私の見解は次のとおりである。

1. **この repo では、feature ごとに「最小の runnable slice」を作るやり方が合っている。**
   - 全文法が確定してから初めて走らせる、という順序にはしない方がよい。
2. **ただし、running sample を増やすことと、language core を広げることは分けるべきである。**
   - settled subset の sample を厚くするのは良い。
   - まだ重い理論線を current executable core に混ぜるのは早い。
3. **formal boundary を並走させるのは正しい。**
   - current repo では tool-neutral formal hook と proof-notebook first bridge まで actualize 済みで、これは良い方向だと考える。
4. **distributed / shared-space / backend は、core semantics と fixed-subset sample の ratchet を壊さない順で入れるべきである。**

## 5. LLVM や backend は、もう前段ならやってよいのか

### 結論

**LLVM-family backend を mainline に入れるのは、まだ早い。**
ただし、その前段の

- fixed-subset source sample
- lowering
- runner
- verification ladder
- theorem / model-check bridge sketch

は、もう進めてよいし、実際に進めている。

### 理由

backend を早く入れると、次が逆流しやすい。

- final grammar
- IR shape
- public runtime API
- host interface
- formal handoff shape

現時点では、これらはまだ narrow に固定したい段階ではない。

一方で、

- sample corpus を増やす
- regression を厚くする
- `e3` のような deferred row を戻す
- proof-notebook / compare-ready bridge を進める

のは、むしろ今やるべき前段である。

したがって current recommendation は、

- **backend は later**
- **その前段の executable / verifiable subset 拡張は now**

### host-facing I/O / FFI / game engine integration について

ここでいう前段には、**docs-first の host-facing I/O / adapter boundary** も含まれる。current repo では、この docs-first boundary 自体は `specs/examples/385...386` で fixed 済みである。

私の current recommendation は次のとおりである。

- `stdin/stdout` を privileged primitive として language core に固定するのは避ける
- 代わりに、capability-scoped な input/output port と adapter boundary を first candidate にする
- visualizer / host substrate / host runtime は、その adapter の consumer/provider 側に置く
- FFI や game engine integration は、そのさらに後段の adapter line に置く

つまり順序としては、

1. fixed-subset sample と theorem/model-check milestone
2. docs-first I/O / host-facing port boundary（fixed 済み）
3. visualizer / host substrate / FFI / engine adapter
4. backend / codegen / public CLI の finalization

が自然である。

ここでの `host-facing port` は settled term ではなく、working label に留めるのがよい。
current docs での anchor は `Typed-Effect Wiring Platform`、`richer host interface`、`C ABI / engine adapters` であり、final naming はまだ OPEN である。

## 6. `atomic_cut` や memory-order 的な制御構造を揃えながら、動く sample を増やしていくのは良いか

### 結論

**半分 yes、半分 no** である。

### 良い部分

settled subset に属するもの、特に

- `atomic_cut`
- `try`
- guarded option chain
- current contract surface

の runnable sample を増やすのは良い。
これは semantics drift を抑え、formal boundary の ratchet にも効く。

### まだ早い部分

次を current executable core に同時投入するのはまだ早い。

- higher-level async-control family
- low-level memory-order-like surface
- fairness / scheduler / distributed control を伴う family

理由は、これらは

- parser
- checker
- runtime
- proof boundary
- shared-space / fabric

の全部へ同時に pressure をかけるからである。

### 私の見解

したがって私は、

- **`atomic_cut` を含む settled subset の sample を厚くするのは賛成**
- **memory-order 再構築 family を current executable core に早く入れるのは反対**

という立場である。

最小 core を壊さずに sample を増やし、その後で高位制御 family を separate track として開く方が、理論も実装も整理しやすい。

## 7. Rust と Python の使い分けをどう考えているか

### 短い答え

**「最終的に全部 Rust に寄せる」とは、現時点では読まない方が正確である。**

より正確には、

- **実行 core / parser / checker / runtime / formal handoff の本体は Rust-heavy に寄る**
- **repo-local な orchestration / regression / scaffolding / docs validation は Python が残ってよい**

という読みが自然である。

### current repo で Rust が持っているもの

- `mir-ast`
  - current L2 parser carrier
- `mir-semantics`
  - parser-free interpreter
  - static gate
  - host harness
  - detached / formal / proof-notebook emitters
- `mir-runtime`
  - source lowerer
  - current L2 source sample runner
  - checker/runtime thin bridge

### current repo で Python が持っているもの

- report / docs validation
- detached-loop orchestration
- source-sample regression helper
- narrow compare/checker helper
- fixture scaffolding

### long-term reading

explicit な docs direction としては、
core implementation は Rust-heavy である。

ただし、そこから直ちに
「repo から Python をなくす」
とは読めない。

むしろ current repo から自然に読めるのは、

- **execution-critical / stable surface は Rust**
- **repo-local helper workflow は mixed tool**

という split である。

### 私の見解

私はこの split が妥当だと考える。

- parser / checker / runtime / formal bridge / public operational CLI まで全部 Python に残すのはよくない
- しかし docs validation / report scaffolding / artifact diff / regression orchestration まで無理に全部 Rust へ寄せる必要もない

つまり、**Rust-heavy core と mixed helper workflow** が current best reading である。
